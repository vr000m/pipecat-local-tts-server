Requirements

ID Module Description
R1 protocol.py

Protocol — PROTOCOL_VERSION="0.1", pcm16 mono, per-backend rate. Full event set per Technical Specifications; ErrorCode enum mirroring stt.

  • hello.audio.rate is the true model rate (Kokoro 24000, confirmed via --load 2026-06-24). MUST be read at connect with no warmup dependency.
  • Every response.audio.delta for the session MUST be int16-LE mono at exactly that rate — no per-utterance drift. Client resamples model-rate → device-rate off this single advertised value.
  • audio_format is strict: only the advertised pcm16-at-hello.audio.rate is accepted; any other value in session.update is rejected with error {code: UNSUPPORTED_FORMAT}.
  • input_text.commit has no audio_format field in v1; if a client sends one it is a protocol error, not format negotiation.
  • ErrorCode adds BUSY (WebSocket analog of HTTP 429): error event carries retry_after_ms when a commit is rejected for synthesis-backlog backpressure (see R4).
R2 backend.py

Backend abstraction — TTSBackend + TTSStream Protocols plus a dependency-free ToneBackend (sine) reference for tests.

  • ToneBackend MUST emit multiple segments with a configurable per-segment delay to drive the per-segment events() streaming contract (R4) and the 20 ms re-chunker — it is the streaming reference, not a one-shot echo.
  • ToneBackend and the shared float→pcm16 converter are stdlib-only runtime code (math/array/struct). numpy may stay in the dev group for verification scripts but MUST NOT be required by runtime modules or lean tests.
R3 backends/kokoro.py

Kokoro backend — mlx-audio load/generate with float→pcm16 conversion.

  • Clip samples to [-1, 1]; map asymmetrically: negative → ×32768, non-negative → ×32767, so -1.0 → -32768 and +1.0 → +32767.
  • Run the generate() generator in a dedicated thread. Metal is not concurrent-safe: use a Lock-pair + in-flight-drain pattern holding the process-wide Metal lock around the entire generator drain (many yields). Lock + per-chunk queue live in backends/_stream_util.py.
  • cancel() MUST break the generator so a cancelled response does not pin the lock.
  • sample_rate is 24000, readable as model.sample_rate immediately after load() — no warmup-generate needed. Warmup at start() is still worthwhile for Metal JIT, but decouple from rate discovery.
  • Lazy-import mlx_audio inside start()/_get_model — never at module load (lean-base invariant).
  • Dependency note: importing Kokoro pulls misaki (G2P) → num2words, spacy, en_core_web_sm, and torch. Keep behind the kokoro extra.
  • Kokoro yields per-segment (split_pattern=r"\n+"); effective streaming granularity is per-segment even though it advertises streaming:false.
R4 server.py

Server/session — handshake (server.hello), per-session text buffer, commit→synthesize, 20 ms re-chunker, response.cancel (barge-in), send-queue high-water close, resource limits.

  • Exact, stable emitted rate: every frame MUST match the advertised hello.audio.rate (per R1).
  • Steady in-response streaming: once response.created fires, feed the client buffer continuously. Do NOT block the send loop on synthesis between chunks; do NOT emit one large burst then a long gap. For Kokoro, emit each segment's audio as it completes.
  • open_stream's end() MUST NOT block until full synthesis while events() replays a stored result — that buffer-then-flush shape is the anti-pattern here.
  • Multi-connection isolation: each connection gets its own _SessionState (text buffer, config, response_id space) — no shared mutable session state. What IS shared: a single model + process-wide Metal lock.
  • Commit = unit of GPU-lock holding: server admits commits into per-connection queues; a single dispatcher selects connections round-robin at commit boundaries; only the selected commit's worker may acquire the Metal lock.
  • Backpressure (two layers):
    • (1) Per-connection outbound send-queue high-water close (slow audio reader).
    • (2) Global synthesis backlog admission control: when the bounded synthesis queue is full the server MUST reject the commit with error {code: BUSY, retry_after_ms}.
    • Per-connection in-flight cap ≤K queued commits. v1 sets K=1 — one active-or-queued response per connection — making response.cancel optional response_id unambiguous.
    • Client retry policy: hold text, retry after retry_after_ms with capped backoff + jitter, give up after 5 retries.
R5 client.py

Client — async TTSClient with transport-generic API (no app labels/frame types — the pipecat adapter lives in examples/).

Methods: connect() → hello, update(), append(), commit(), cancel(), events(), status(), close().
R6 __main__.py

CLI — mirroring stt.

  • python -m tts_server serve --backend kokoro --model … --socket-path … (logs resolved backend+model at startup).
  • status health probe: connect → hello → status → print backend/model/rate/queue depth.
R7 protocol.py

Capabilities for client chunking — capabilities MUST expose: streaming, ideal_words, text_formats, languages, extras, max_text_chars.

  • ideal_words is a soft chunk-size target; the client rounds up to the next sentence boundary (never split mid-sentence — decision #6).
  • max_text_chars is the hard server cap (reject beyond it).
  • For streaming:false backends the client MUST chunk at sentences.
  • Unknown extras keys are dropped (debug-logged), never errored.
  • extras per backend (verified via scripts/verify_mlx_tts_api.py against mlx-audio 0.4.4):
    • Kokoro → {speed} only
    • voxtral_tts → {temperature, top_k, top_p}
    • pocket_tts → {temperature}
    • dia → {temperature, top_p}
  • A backend MUST drop, not forward, any kwarg the model ignores — the advertised extras never lies to the client.
R8 pyproject.toml

Packaging — package pipecat-local-tts-server, import tts_server.

  • Lean base = websockets only.
  • Extras: client, kokoro (+ later voxtral_tts, pocket_tts, dia/chatterbox). All backends lazy-import heavy deps.
  • Pin mlx-audio==0.4.4 in backend extras — TTS API drifted between 0.3.0 and 0.4.4. Re-run scripts/verify_mlx_tts_api.py before widening the pin.
